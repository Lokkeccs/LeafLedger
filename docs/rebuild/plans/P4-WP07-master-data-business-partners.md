## Context / scope note (LL Architect)

Per the [feature roadmap](../architecture/rebuild/06-feature-roadmap.md) M1 **Master data** line: *"Users, business partners (+accounts/groups), projects, currencies — CRUD + import/export."* WP07 delivers the **business-partners CRUD slice**: create / read / edit / activate-deactivate / delete of business partners, server-authoritative, RLS-scoped, audited, idempotent, with optimistic concurrency — plus the management UI. The remaining master-data breadth is carved to adjacent WPs: **projects + currencies → P4-WP08**; **journal partner links + posting integration → P4-WP09**; **partner default-account mappings** and **master-data import/export** are deferred (see Non-goals). This keeps WP07 ≤ 2 days and free of un-consulted accounting decisions.

This is a **rewrite** WP per §5: the OLD business-partners page mutated Dexie via a `DataApi` (`businessPartnersDataApi.saveBusinessPartner`/`deleteBusinessPartner` over `data/db.ts`), so *"all data access Dexie → TanStack Query + generated client"* — the OLD read/write path is **not ported**; only the field set and page shape inform the new surface. The OLD `BusinessPartner` shape (`data/db.ts` L121–132) is minimal reference data: `name`, `type ∈ {customer, vendor, both, financial-services}`, `isActive?`, `validFrom?`, `validTo?` — there is **no accounting engine** here, so there are no golden fixtures and no accounting consult (see below).

**Departure from the coarse forward-map note.** The Phase-4 tracker row said *"New Master-data module + tables."* On grounding against merged code, a dedicated `LeafLedger.Modules.MasterData` module would introduce a **second module DbContext** (the spec's *"one DbContext per module"* rule) and duplicate the RLS/authz/idempotency/audit/migration plumbing for a trivial CRUD table. Because `journal_lines.business_partner_id` is Ledger-owned and the `PostingValidity` domain **already** validates business-partner references (`PostingValidityEvaluator.AssertPostingBusinessPartnersValid`, `PostingEntityType.BusinessPartner`), the natural, lowest-risk home is the **existing Ledger module** — identical to **D-P3-ACCT-MODULE** (accounts endpoints/EF live in Ledger Infrastructure while ChartOfAccounts stays a pure domain module). This is surfaced as **D-P4-BP-MODULE** for explicit sign-off, since it also sets the pattern for P4-WP08.

## Backend design (WP07a — the business-partners CRUD endpoints)

**New EF entity + one new migration** (unlike P4-WP05, which reused existing tables). The migration follows the InitialLedgerSchema DDL patterns verbatim.

- **EF entity — `Infrastructure/Entities/BusinessPartner.cs`:** `Id (uuid) `, `SpaceId (uuid)`, `Name (text)`, `PartnerNumber (text?)`, `Type (text)`, `CountryCode (text?)`, `IsActive (bool)`, `ValidFrom (date?)`, `ValidTo (date?)`, `Notes (text?)`, `CreatedBy (uuid)`, `CreatedAt (timestamptz)`, `UpdatedAt (timestamptz)`, and an **xmin concurrency token** mapped via Npgsql `.UseXminAsConcurrencyToken()` (a Postgres **system column** — no DDL, purely EF config). The three D-P4-BP-FIELDS columns are **inert** (no behavior, no FK, no CHECK beyond shape). Registered in `LedgerDbContext` `OnModelCreating` with `space_id` FK, `UNIQUE(space_id, name)`, a partial `UNIQUE(space_id, partner_number)` (D-P4-BP-FIELDS), and snake-case column names.
- **Migration — `{timestamp}_BusinessPartners`:** `CREATE TABLE business_partners (…)` via `migrationBuilder.CreateTable` (EF fluent, so `HasPendingModelChanges()` stays green), plus raw `migrationBuilder.Sql` for the three cross-cutting concerns exactly as the initial schema does:
  1. audit trigger `CREATE TRIGGER trg_business_partners_audit AFTER INSERT OR UPDATE OR DELETE ON business_partners FOR EACH ROW EXECUTE FUNCTION write_audit_log();`
  2. grants `GRANT SELECT, INSERT, UPDATE, DELETE ON business_partners TO leafledger_app;` (master data is mutable — SELECT/INSERT/UPDATE/DELETE, unlike append-only journal tables);
  3. RLS `ALTER TABLE business_partners ENABLE ROW LEVEL SECURITY; … FORCE …; CREATE POLICY p_business_partners_isolation … USING/WITH CHECK (space_id = current_setting('app.current_space_id', true)::uuid);`.
  The table also carries `partner_number text NULL`, `country_code text NULL`, and `notes text NULL`, with a partial `UNIQUE(space_id, partner_number) WHERE partner_number IS NOT NULL` (D-P4-BP-FIELDS); `country_code` has no DB-level CHECK (shape validated in the service, kept inert for the VAT WP).
  A reversible `Down` drops the trigger, policy, and table.
- **Write service (Ledger Infrastructure) — `IBusinessPartnerService` + `BusinessPartnerService`:** mirrors `AccountManagementService` — open connection, begin transaction, `BindTransactionAsync(connection, txn, spaceId, actorId)` (principal-bound RLS + `app.current_actor` for the audit trigger), enforce/replay/store idempotency inside the transaction, execute, commit, enqueue space invalidation. Methods (M1):
  - `CreateAsync(spaceId, actor, idemKey, CreateBusinessPartnerCommand)` — validate `name` non-empty, `type ∈ enum`, `valid_from ≤ valid_to`, `partner_number` (if provided) unique per space, `country_code` (if provided) a 2-letter ISO 3166-1 alpha-2 **shape** (not a VAT-relevance check — D-P4-BP-FIELDS); insert; `UNIQUE(space_id, name)` violation → `422 partner.name_taken`, `UNIQUE(space_id, partner_number)` violation → `422 partner.number_taken`, malformed country → `422 partner.country_invalid`.
  - `UpdateAsync(…, partnerId, expectedVersion, UpdateBusinessPartnerCommand)` — same field validation + **optimistic concurrency**: the update is guarded by the client-supplied `xmin` token; a stale token → `409 partner.version_conflict` with the **current** partner state in the response body (§4.4). `is_active` is edited here (D-P4-BP-VERBS — no post-posting immutability, so a single `PATCH` suffices).
  - `DeleteAsync(…, partnerId)` — delete when unreferenced; if any `journal_lines.business_partner_id = partnerId` exists → `409 partner.in_use` (explicit reference-count query, correct even before the FK constraint lands in P4-WP09 — D-P4-BP-FK). `is_active=false` remains the reversible soft-hide alternative.
- **Read (RLS-bound, `ledger.read`):** `GET …/partners` → `BusinessPartnerCatalogReport` (list, ordered by name) and `GET …/partners/{id}` → single `BusinessPartnerView` (both carry the `xmin` version token so the edit form can round-trip it). Mirrors the P3-WP05 read shape.
- **Idempotency (D-P4-BP-IDEM):** create / update / delete require an `Idempotency-Key` (ULID) header, enforced with the same `TryGetIdempotencyKey` validation (`400 idempotency.key_required`/`key_invalid`), request-hash replay, and advisory lock as posting/CoA writes (generalize the envelope `target` to cover `partner.create`/`partner.update`/`partner.delete`). Retried same key+payload → original 2xx replayed with `Idempotent-Replayed: true`; same key+different payload → `409 idempotency.key_reused`.
- **Contracts (`Application/MasterData/BusinessPartnerContracts.cs`):** `CreateBusinessPartnerCommand`/`UpdateBusinessPartnerCommand` (both carry `Name`, `Type`, `IsActive`, `ValidFrom?`, `ValidTo?`, and the D-P4-BP-FIELDS additions `PartnerNumber?`, `CountryCode?`, `Notes?`), `BusinessPartnerView(Guid Id, string Name, string? PartnerNumber, string Type, string? CountryCode, bool IsActive, DateOnly? ValidFrom, DateOnly? ValidTo, string? Notes, string Version)`, `BusinessPartnerCatalogReport`, `IBusinessPartnerService`.
- **Routes (extend `LedgerEndpoints`, existing `configureAuthorization` seam):**
  - `GET    …/partners` → 200 `BusinessPartnerCatalogReport` (`ledger.read`).
  - `GET    …/partners/{id:guid}` → 200 `BusinessPartnerView` / 404 (`ledger.read`).
  - `POST   …/partners` → 201 `BusinessPartnerView` (`partners.manage`, Idempotency-Key, 400/409/422/401/403).
  - `PATCH  …/partners/{id:guid}` → 200 `BusinessPartnerView` (`partners.manage`, idempotent, `409 partner.version_conflict` on stale xmin).
  - `DELETE …/partners/{id:guid}` → 204 (`partners.manage`, idempotent, `409 partner.in_use`).
  All 422/409 use stable ProblemDetails codes + `extensions["errors"]` `{ code, message, field? }`, exactly like posting/CoA writes.
- **Permission (`ModulePermissions`) — new `partners.manage` (D-P4-BP-PERM):** granted to **Owner + Admin + Member** (business partners are everyday operational reference data used while posting — unlike structural chart maintenance, which is Owner/Admin). Viewer keeps read-only via `ledger.read`. Extend `Allows` + `IsKnownPermission` + the WP06 authorization tests.
- **OpenAPI/TS regeneration (REQUIRED):** rebuild `backend/openapi/leafledger-v1.json` + regenerate `app/src/api/schema.d.ts`; the CI `contract` gate must be green. A regen diff here is **expected and required**.
- **Register `IBusinessPartnerService` in `LedgerModule.cs`.**

## Goal

An authorized member (Owner/Admin/Member) can **manage business partners from the app** — create a partner (name, optional partner-number, type, optional country, active, validity, optional notes), edit it (with optimistic-concurrency protection), activate/deactivate it, and delete it when it is not referenced by any posting — every write **server-authoritative**, idempotent, RLS-scoped, and audited, with structured 422/409 rendered inline. Concretely: WP07a creates the `business_partners` table (EF entity + migration with RLS/audit/grants) and exposes the CRUD endpoints under a new `partners.manage` permission with idempotency, xmin optimistic concurrency, in-use delete-protection, and OpenAPI/TS regen; WP07b renders the management UI (list + form modal + delete/activate) over TanStack Query with UX-only mirrored validation and live invalidation.

This WP adds **no** journal/posting integration (P4-WP09), **no** partner default-account mappings, **no** projects/currencies (P4-WP08), **no** master-data import/export, **no** `journal_lines` FK constraint, and **no** client-side role/rights table.

## Decisions (front-loaded — approved 2026-07-24)

All are **non-accounting** engineering decisions; the user **approved every recommendation on 2026-07-24**. **D-P4-BP-FIELDS** (below) was added after an LL Accounting Expert field-set consult and is likewise approved.

- **D-P4-BP-MODULE — home for `business_partners`.** **RECOMMEND: the existing Ledger module** (entity/migration in Ledger Infrastructure; endpoints in `LedgerEndpoints`), superseding the coarse *"new Master-data module"* forward-map note. Rationale: single DbContext (D1) avoids a second migration pipeline; `journal_lines.business_partner_id` is Ledger-owned and `PostingValidity` already validates partner references; identical to D-P3-ACCT-MODULE. *Alternative:* a new `LeafLedger.Modules.MasterData` module + DbContext — rejected for M1 (over-engineered for a trivial CRUD table; duplicates plumbing; a second DbContext is a meaningful architectural commitment better made when a real master-data domain justifies it). This decision also sets the P4-WP08 pattern.
- **D-P4-BP-SPLIT — one WP or two.** **RECOMMEND: split WP07a (backend) → WP07b (frontend)** (frontend types come from the regenerated OpenAPI; combined effort exceeds ≤ 2 days). *Alternative:* one combined WP — rejected (over the ceiling; hard frontend→backend regen dependency).
- **D-P4-BP-PERM — authorization for writes.** **RECOMMEND: a new `partners.manage` permission granted to Owner + Admin + Member** (business partners are everyday operational reference data used while posting; restricting them to admins would block ordinary members from recording a counterparty). Reads stay `ledger.read` (all roles incl. Viewer). *Alternative A:* Owner/Admin only (mirror `accounts.manage`) — rejected for M1 (partners are operational, not structural). *Alternative B:* reuse `ledger.post` — rejected (posting ≠ master-data maintenance). Revisit granularity in Phase-4 admin (P4-WP18).
- **D-P4-BP-CONCURRENCY — concurrent-edit model.** **RECOMMEND: optimistic concurrency via the Postgres `xmin` system column** (§4.4): reads return a `version` token, `PATCH` requires it, a stale token → `409 partner.version_conflict` with current state; the frontend does a **refetch-and-retry** (no complex merge UI at M1). Rationale: this is the spec's stated master-data model and establishes the reusable pattern for P4-WP08. *Alternative:* last-write-wins + idempotency only, defer xmin — rejected (silently loses concurrent edits; contradicts §4.4; the pattern is cheap now via `.UseXminAsConcurrencyToken()`).
- **D-P4-BP-IDEM — idempotency for master-data writes.** **RECOMMEND: reuse the P2-WP09 envelope** for create/update/delete (generalize `target` to `partner.*`), keeping the required-header contract. *Alternative:* skip idempotency — **rejected** (violates non-negotiable #4).
- **D-P4-BP-VERBS — HTTP shape for edit + activate/deactivate.** **RECOMMEND: a single `PATCH …/partners/{id}` carries all mutable fields including `is_active`** (partners have no post-posting immutability and use xmin concurrency, so the WP05 sub-resource-verb rationale does not apply). Create = `POST`, delete = `DELETE`. *Alternative:* sub-resource `activate`/`deactivate` verbs (WP05 style) — acceptable but heavier; not recommended for this entity.
- **D-P4-BP-DELETE — delete semantics.** **RECOMMEND: full `DELETE` allowed only when the partner is unreferenced by any `journal_line` (→ `409 partner.in_use`), plus `is_active=false` as the reversible soft-hide.** In M1 nothing references partners yet (linkage is P4-WP09), so delete is trivially safe now, but the guard exists for forward-safety and the roadmap's "CRUD" includes delete. *Alternative:* deactivate-only, defer hard delete — rejected (roadmap wants CRUD; the in-use guard is a cheap reference-count query).
- **D-P4-BP-FK — `journal_lines` FK constraint.** **RECOMMEND: defer the `journal_lines.business_partner_id → business_partners(id)` FK constraint (and posting wiring) to P4-WP09.** WP07 delete-protection uses an explicit reference-count query, correct even before the FK exists, and WP07 does not touch the append-only `journal_lines` table. *Alternative:* add the nullable FK now — acceptable (gives DB-level in-use protection) but couples WP07 to the append-only-table grants and pre-empts P4-WP09's integration; not recommended.
- **D-P4-BP-UIGATE — client capability gating.** **RECOMMEND: render affordances, rely on the server `partners.manage` 403 as the wall** (friendly message on 403), reusing D-P4-COA-UIGATE. *Alternative:* client role lookup — rejected (re-introduces the OLD client rights table).
- **D-P4-BP-FIELDS — business-partner field set (LL Accounting Expert consulted 2026-07-24).** **APPROVED: extend the OLD five fields (`name`, `type`, `is_active`, `valid_from`, `valid_to`) with three inert, non-behavioral descriptive columns** — `partner_number` (optional external reference, `UNIQUE(space_id, partner_number)` when present), `country_code` (optional ISO 3166-1 alpha-2, **stored but not acted upon**), and `notes` (optional free text). **Structural validation only:** `partner_number` unique → `422 partner.number_taken`; `country_code` must be a 2-letter ISO shape if provided → `422 partner.country_invalid` (a shape check, **not** a VAT-relevance check). *Rationale (accounting expert):* a partner in M1 is only an identity to attach to journal lines; the accounting-load-bearing attributes are all VAT/FX-driven and must not be stored before their rule + golden fixtures exist (a dangling tax/account field invites behavior wired without a fixture). `country_code` is the single most accounting-relevant attribute (domestic vs foreign → VAT/export), so it is front-loaded as **inert data** for the VAT WP to consume; `partner_number`/`notes` are operational conveniences. Because nullable columns are additive and cheap in Postgres/EF, **everything behavioral is deferred with no migration penalty** (see the deferred-fields carry-forward). *Alternative:* add UID/VAT number, tax status, default accounts, currency, or payment terms now — **rejected** (each implies accounting behavior needing an LL Accounting Expert consult + golden fixtures in its owning WP; storing them inert here is a correctness hazard).

## Accounting decisions

**LL Accounting Expert consulted 2026-07-24 (business-partner field-set guidance) — verdict: no accounting behavior enters WP07, no golden fixtures required.** Business partners are **reference master data** (name, categorization label, validity window, and the inert descriptive fields of D-P4-BP-FIELDS), not an accounting rule or computation: money is not involved, no posting/VAT/FX behavior is decided here, and validation is purely structural (name required, type in enum, `valid_from ≤ valid_to`, name + `partner_number` unique per space, `country_code` a 2-letter ISO shape). The `type` enum (`customer|vendor|both|financial-services`) and `country_code` are **informational/inert** — ported/stored verbatim and **not** acted upon for posting/VAT in M1.

**Field-set guidance (recorded).** The expert's tiered recommendation: (a) **add now** only inert identity/descriptive fields (`partner_number`, `country_code`, `notes`) — `country_code` front-loaded because it is the single most VAT-relevant partner attribute (domestic vs foreign/export), stored but not acted upon; (b) **defer all accounting-behavioral attributes** (UID/VAT number, VAT liability/tax status, default tax code, default accounts, default currency, payment terms, IBAN, full address) to their owning rule WPs, each gated by a consult + golden fixtures — additive nullable columns make deferral penalty-free. Recorded as a carry-forward so P4-WP09/P4-WP13 revisit partner-type semantics if VAT/reporting keys off it. Money stays integer minor units + ISO currency (not applicable here — partners carry no amounts). The client mirrors server validation for UX only; the server + DB constraints are the wall.

## Acceptance criteria (concrete, testable)

1. **Create partner.** `POST …/partners` with a valid command persists the partner (RLS-scoped, audited) and returns 201 `BusinessPartnerView` (with a `version` token); an integration test asserts the round-trip and that the audit trigger wrote a row.
2. **Structural validation.** Duplicate name → `422 partner.name_taken`; duplicate `partner_number` → `422 partner.number_taken`; invalid `type` → `422 partner.type_invalid`; malformed `country_code` (non-ISO-alpha-2) → `422 partner.country_invalid`; `valid_from > valid_to` → `422 partner.validity_range_invalid`. `country_code` is validated for **shape only** (no VAT-relevance logic — D-P4-BP-FIELDS). All as stable ProblemDetails codes with `extensions["errors"]`.
3. **Edit + optimistic concurrency (§4.4).** `PATCH …/partners/{id}` with the current `version` updates mutable fields (incl. `is_active`) and writes an audit row; a `PATCH` with a **stale** `version` returns `409 partner.version_conflict` carrying the current partner state, and the stored row is unchanged.
4. **Activate/deactivate.** Setting `is_active=false` via `PATCH` hides the partner from active lists; setting it back restores it; idempotent (re-applying is a no-op success).
5. **Delete + in-use guard.** `DELETE …/partners/{id}` on an unreferenced partner returns 204; on a partner referenced by a seeded `journal_line.business_partner_id` returns `409 partner.in_use` and leaves the row intact.
6. **Reads + RLS.** `GET …/partners` returns the space's partners ordered by name (each with a `version`); `GET …/partners/{id}` returns one / 404; a second space's partners are invisible under RLS (two-space test).
7. **Authorization (`partners.manage`).** Every write is `partners.manage`-gated: anonymous → 401, a Viewer → 403 on writes (200 on reads), Member/Admin/Owner → 2xx on writes; cross-space writes are blocked by the RLS second wall.
8. **Idempotency (non-negotiable #4).** Create/update/delete require a valid ULID `Idempotency-Key` (`400 idempotency.key_required`/`key_invalid` otherwise); a retried create with the same key+payload replays exactly one row with `Idempotent-Replayed: true`; same key + different payload → `409 idempotency.key_reused`.
9. **Schema + no-drift.** The `business_partners` table exists with the RLS policy, audit trigger, grants, and `UNIQUE(space_id, name)`; the migration `Down` is reversible; `HasPendingModelChanges()` no-drift test stays green (entity + migration in sync).
10. **Contract regenerated and gate-green.** `backend/openapi/leafledger-v1.json` + `app/src/api/schema.d.ts` regenerated and committed; CI `contract` gate green; the five operations + new schemas present.
11. **Management UI.** An authorized member can create/edit a partner and delete/deactivate one through modal forms rendering shared primitives, and see inline server 422/409 (incl. version-conflict "reload" and in-use messages); loading/submitting states present; each page/modal ≤ 450 lines (page budget green) — proven by component tests.
12. **Layering + query conventions + invalidation.** UI reads/writes only through `application`; hooks use `qk.businessPartners.list` and invalidate it on success; no direct `fetch`/`src/api` import in `features`; ESLint boundary/layering green.
13. **i18n + no out-of-scope surface.** All partner-management copy in EN + DE (duplicate-key gate green); no journal/posting integration, no `journal_lines` FK, no partner-account mappings, no projects/currencies, no import/export, no client rights table; a scope scan confirms the diff is limited to the file list.
