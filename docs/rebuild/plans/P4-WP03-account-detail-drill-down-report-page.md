## Context / scope note (LL Architect)

The [feature roadmap](../architecture/rebuild/06-feature-roadmap.md) M1 Reports row lists *"Trial balance, balance sheet, P&L, account detail/drill, overview dashboard (server-computed)"* (line 44); *"drill-everything"* and pivots are explicitly M2. P3-WP07 shipped the trial balance; P4-WP02 shipped the balance sheet + P&L. **This WP is the "account detail/drill" entry** — the per-account posted-lines view. The overview dashboard is the remaining M1 report (P4-WP04).

This is the **first Phase-4 report whose data does not already exist server-side**: the P2-WP07 views expose *aggregated* per-account balances (trial balance) and *presentation* statements (BS/P&L), but there is **no read of the individual posted `journal_lines` for one account** anywhere. So WP03 must add the read. Per Part 4 §5 this is a **spec-derived rewrite / new server capability (§72)** — the OLD account-detail drill was UI-computed over Dexie/IndexedDB (same character as the OLD BS/P&L: no capturable pure oracle, and the derived cached balances were exactly the drift the server-side approach eliminates), so it is **not a port**; only the abstract shape (a date-ordered line list with a running balance) informs the new endpoint.

**Key structural finding — no new view / no migration.** A per-account, **date-scoped**, running-balance read is naturally a **parameterized query** (params: `accountId`, optional `from`/`to` on `journal_entries.date`, plus the opening/brought-forward figure), which a plain `security_invoker` *view* cannot express cleanly (a view cannot take the date window or the brought-forward parameter). The recommended route is therefore a **parameterized SQL read bound through the existing RLS pattern** in a small read service — **no new SQL view, no EF entity, no migration** (the `SELECT` grants + RLS policies on `journal_lines`/`journal_entries`/`accounts` already exist from P2-WP02, so `HasPendingModelChanges()` stays green). This is the mechanically simplest way to satisfy "posted lines for one account, date-scoped" (D-P4-AD-BACKEND-SHAPE).

**The one accounting-adjacent point — the running balance.** A raw list of the posted lines is a pure relational projection that computes no accounting value (needs neither fixtures nor a consult). The moment the drill shows a **running balance** with a date window, two presentation-accounting notions appear: (1) the single-account running-balance **sign convention**, and (2) whether the running balance starts at **zero** or at the account's **opening balance brought forward** (the cumulative posted amount strictly before the window). That is the only place accounting semantics can enter; it is front-loaded as D-P4-AD-RUNNINGBALANCE with an explicit route determination for the golden-fixture / accounting-consult question (see those sections).

## Backend design (WP03a — new, this WP adds it)

New read, **no new view / no migration** (parameterized RLS-bound query; D-P4-AD-BACKEND-SHAPE).

- **Route:** `GET /api/v1/spaces/{spaceId:guid}/reports/account-ledger/{accountId:guid}?from={date}&to={date}` — registered on the existing `reportGroup` in `LedgerEndpoints`, `.WithName("GetAccountLedger")`, `.Produces<AccountLedgerReport>(200)` + 401/403 ProblemDetails, authorized with **`ledger.read`** via the existing `configureAuthorization` seam (D-P4-AD-PERM). Read-only; no idempotency header (GET). `from`/`to` are **optional** ISO dates on `journal_entries.date` (inclusive `from`, inclusive `to`); both omitted ⇒ full history (D-P4-AD-DATESCOPE). Errors: `401 auth.unauthenticated`, `403 auth.not_a_member`/`auth.permission_denied`; an unknown/foreign `accountId` returns `200` with an empty `lines` list and zeroed balances (RLS + `WHERE account_id` naturally scope it — never 404/leak).
- **Read service (Ledger Infrastructure):** `IAccountLedgerService` + `AccountLedgerService` mirroring `AccountCatalogService`/`LedgerReportService` — one method `Task<AccountLedgerReport> GetAccountLedgerAsync(Guid spaceId, Guid accountId, DateOnly? from, DateOnly? to, CancellationToken)` using the **same `OpenBoundConnectionAsync` txn-local RLS binding** (rollback-on-dispose). Two parameterized statements against the RLS-bound connection:
  1. the **posted lines** — `SELECT je.id, je.entry_no, je.date, je.description, je.reference, jl.amount_minor, jl.base_amount_minor, jl.currency FROM journal_lines jl JOIN journal_entries je ON je.id = jl.entry_id WHERE jl.account_id = @account AND je.status = 'posted' AND (@from IS NULL OR je.date >= @from) AND (@to IS NULL OR je.date <= @to) ORDER BY je.date, je.entry_no, jl.id;` with a window running balance `SUM(jl.base_amount_minor) OVER (ORDER BY je.date, je.entry_no, jl.id)` added to the opening figure;
  2. the **opening balance brought forward** (only when `@from` is set) — `SELECT COALESCE(SUM(jl.base_amount_minor),0) FROM journal_lines jl JOIN journal_entries je ON je.id = jl.entry_id WHERE jl.account_id = @account AND je.status = 'posted' AND je.date < @from;` (per D-P4-AD-RUNNINGBALANCE / consult C-P4-AD-LEDGER).
  The account header fields (`code`, `name`, `kind`, `currency`) come from a single `accounts` lookup under the same binding.
- **Contract (`Application/Reporting/…` records, `LedgerReportContracts` style):**
  - `AccountLedgerLine(Guid EntryId, long EntryNo, DateOnly Date, string? Description, string? Reference, long AmountMinor, long BaseAmountMinor, string LineCurrency, long RunningBalanceMinor)`.
  - `AccountLedgerReport(Guid SpaceId, Guid AccountId, int AccountCode, string AccountName, string AccountKind, string AccountCurrency, long OpeningBalanceMinor, long ClosingBalanceMinor, IReadOnlyList<AccountLedgerLine> Lines)`.
  - All amounts are **integer minor units** in the raw debit-positive convention (matching the trial balance); no float. `OpeningBalanceMinor` is `0` when `from` is absent; `ClosingBalanceMinor` = opening + Σ line base amounts (== the trial-balance `base_balance_minor` for the account when the window is full history).
- **OpenAPI/TS regeneration (REQUIRED — the differentiator from P4-WP02):** rebuild `backend/openapi/leafledger-v1.json` and regenerate `app/src/api/schema.d.ts` so the frontend gets typed `AccountLedgerReport`/`AccountLedgerLine` and the `GetAccountLedger` operation. The CI `contract` diff gate must be green (committed artifacts match generated). *(In P4-WP02 a regen diff meant "stop — unexpected backend change"; here a regen diff is expected and required.)*
- **No EF model change:** raw-SQL parameterized read; no entity, no migration; `HasPendingModelChanges()` no-drift test stays green.

## Goal

A signed-in user opens an account's ledger — by clicking an account in the trial balance / accounts list, or picking it in a selector — and sees the individual posted journal lines that make up that account's balance: date, entry number, description/reference, the transaction amount, the base amount, and a running balance, optionally narrowed to a date window, served by a **new** authorized, RLS-bound backend read. Concretely: WP03a exposes `GET …/reports/account-ledger/{accountId}?from&to` → `AccountLedgerReport` (RLS-bound, `ledger.read`, parameterized query, no new view/migration, OpenAPI/TS regenerated); WP03b renders it as `AccountLedgerPage` through `DataTable` with account selection, date scoping, loading/empty/error states, and post→see-it/live invalidation.

This WP adds **no** overview dashboard (P4-WP04), **no** cursor pagination, **no** VAT/attribution/partner/project columns, **no** export, **no** journal-entry *list* endpoint (`GET journal-entries?filter…` stays deferred), and **no** editing of posted lines.

## Accounting decisions

**C-P4-AD-LEDGER — APPROVED 2026-07-18.** The single-account running-balance sign convention is the **raw debit-positive** convention (same as the trial balance / P2-WP07 C1 raw side — *not* the balance-sheet/P&L presentation flips), so a debit accumulates positive and a credit negative. The **opening balance brought forward** is the cumulative posted `base_amount_minor` strictly before the window `from`; over full history, the closing balance reconciles to the trial-balance `base_balance_minor` for that account. **Reversals need no special handling** — a reversal is itself a posted line and nets naturally into the running balance. This is a presentation rule over existing posted data, not a new accounting engine; pin it with explicit-value integration tests, not golden fixtures.

**Standing guard (from P3-WP07/P4-WP02, load-bearing here):** the client renders server-provided amounts **verbatim** and performs **no** sign flip, abs, aggregation, or re-derivation. The running balance and opening balance are computed **server-side** (in the parameterized query), never in the component. If any presentation rule cannot be traced to the server output or the C-P4-AD-LEDGER confirmation, **stop and route to LL Accounting Expert**. Two boundaries are respected, not decided: money stays integer minor units + ISO currency, formatted only at the render edge (no float math on amounts); the drill is read-only (posted lines immutable, no mutation).

## Acceptance criteria (concrete tests — all must pass)

**WP03a — backend**
1. **Endpoint returns the account's posted lines, ordered.** An integration test seeds an account with ≥ 3 posted lines across ≥ 2 entries; `GET …/reports/account-ledger/{accountId}` returns `AccountLedgerReport` with the lines in `date, entry_no, id` order, each carrying `entryId/entryNo/date/description/reference/amountMinor/baseAmountMinor/lineCurrency`, and the account header (`accountCode/accountName/accountKind/accountCurrency`).
2. **Running balance & closing balance (if D-P4-AD-RUNNINGBALANCE recommended route).** The running balance accumulates `base_amount_minor` in the seeded order; `closingBalanceMinor` == the trial-balance `base_balance_minor` for that account over full history. *(If the lean route is chosen, this AC is replaced by: no running/opening/closing fields are emitted.)*
3. **Opening balance brought forward (recommended route).** With `?from=` set mid-history, `openingBalanceMinor` == Σ posted `base_amount_minor` strictly before `from`, and the first window line's running balance == opening + that line.
4. **Date scoping.** `from`/`to` include their boundary dates and exclude lines outside the window; both omitted ⇒ full history; a window with no lines ⇒ empty `lines` with `openingBalanceMinor` = brought-forward (recommended) / `0` (lean) and `closingBalanceMinor` reconciling.
5. **Reversal nets naturally.** A reversal entry's line appears as its own posted line and the running/closing balance reflects it (no special-casing).
6. **Tenancy (RLS second wall).** A second space's/account's lines never appear in space A's request (two seeded spaces).
7. **Authorization.** Anonymous → 401; a principal without `ledger.read` → 403; a **Viewer** member → 200 (reusing the WP06 auth harness).
8. **Empty / foreign account.** An account with no posted lines, and an `accountId` not in the space, each return `200` with an empty `lines` list and zeroed balances (never 404/500/leak).
9. **No schema/model drift + contract regenerated.** No EF entity/migration; `HasPendingModelChanges()` no-drift stays green; `backend/openapi/leafledger-v1.json` + `app/src/api/schema.d.ts` regenerate and are committed; the CI `contract` gate is green; the new `GetAccountLedger` path + `AccountLedgerReport`/`AccountLedgerLine` schemas are present.
