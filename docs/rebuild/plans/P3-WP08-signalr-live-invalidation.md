## Context / scope note (LL Architect)

Part 4 §Phase 3 exit criterion: *"post a balanced entry → see it in the trial-balance report; two browsers in one space live-update via a SignalR invalidation ping."* WP06+WP07 delivered the first clause end-to-end (post → the invalidated `reports.trialBalance` key refetches **in the posting tab**). WP08 delivers the second clause: a **server-emitted invalidation ping** fans out to every other client connected to that space, each of which refetches through TanStack Query — so browser B's trial-balance page updates seconds after browser A posts, with **no data on the socket** (Part 3 §2 verdict: *"demoted to cache-invalidation pings — no data over the socket"*).

This is a **spec-derived rewrite, not a port** (Part 3 §2: *"Self-hosted SignalR + in-memory tracker → **Replace → Azure SignalR Service**; scale-out safe; demoted to cache-invalidation pings"*). The OLD system had a self-hosted SignalR hub `/hubs/sync` broadcasting `SyncRequired` to space groups with **no backplane** and an **in-memory membership tracker that silently breaks on scale-out to 2 instances** (Part 1 §Realtime; Part 2 §weakness 8). The rewrite keeps only the *abstract shape* (per-space group → invalidation signal → client refetch) and replaces the broken transport with Azure SignalR Service (scale-out safe) plus disciplined **N5 server coalescing** and an **N6 client invalidation map**. There is **no accounting behavior, no financial value, and no data payload** in this WP — it is cache-invalidation transport. Hence **no golden fixtures and no accounting consult** (see those sections).

The two risk-review items this WP owns (both from `NOTES-risk-review-2026-07-06.md`):

- **N5 — SignalR invalidation coalescing** (→ this WP): server-side coalescing at most one ping per space/topic per ~200 ms window; client-side batching debounced per convention (a single utility, not per-page). *Acceptance wording to honor:* "Server test: N rapid postings in one space produce ≤ ceil(window-count) pings, not N. Client test: burst of pings triggers one refetch cycle per query key." Postgres NOTIFY/LISTEN → SignalR was **assessed and rejected as premature — do not implement.**
- **N6 — TanStack Query invalidation map** (→ this WP; foundation laid in WP01's query-key factory): make the module → mutation → query-keys map an **explicit, reviewable deliverable**, not folklore, with cache-correctness tests. *Acceptance wording to honor:* "Invalidation map document exists and covers every mutation in the WP's scope. At least one cache-correctness test per module proving mapped-key refetch + unmapped-key stability."

## Backend seam (already merged — WP08a extends it, does not replace it)

`JournalPostingService.ExecuteAsync` (Ledger Infrastructure) already, **after** `await transaction.CommitAsync(...)`, calls `_refreshQueue.TryEnqueue(spaceId)` (P2-WP12). WP08a adds a **parallel** enqueue on the same line — `_invalidationQueue.TryEnqueue(spaceId, topics)` — feeding a coalescing `BackgroundService` that broadcasts one ping per space/topic per debounce window over SignalR. The ping fires **only after a committed post/reversal** (never inside the transaction, never on a failed/replayed-collision post). No REST contract, OpenAPI, or migration change; the socket is a side channel, not an endpoint in the OpenAPI document.

## Goal

After a signed-in user in browser A posts a balanced entry to a space, every other client connected to that same space (browser B, another tab) refetches the affected TanStack Query keys and updates within a couple of seconds — driven by a single server invalidation ping carrying only `{ spaceId, topic }` (no data), with server-side coalescing so a burst of posts yields at most one ping per space/topic per ~200 ms, and a single client-side invalidation utility that maps a ping topic to its query keys and batches the refetch. Concretely:

1. **Backend (WP08a):** a membership-authorized SignalR hub (`/hubs/space`), a per-space/topic-coalescing post-commit `BackgroundService` (mirroring the WP12 refresh coalescer) that broadcasts `spaceInvalidated({ spaceId, topic })` to the `space:{spaceId}` group, an enqueue at the existing post-commit seam, a coalescing metric, and the Azure SignalR Service config seam (Default mode) for production scale-out.
2. **Frontend (WP08b):** a single `useSpaceInvalidation(spaceId)` hook (mounted once in the shell) that owns one authenticated `HubConnection`, an explicit **N6 invalidation map** (`topic → (spaceId) => queryKey[]`) documented and tested, and batched `queryClient.invalidateQueries` (one refetch cycle per mapped key per burst).
3. **Exit proof (within WP08):** deterministic tests prove "two browsers in one space live-update" — a backend two-`HubConnection` integration test (both clients in one space receive the coalesced ping; a client in another space does not) plus a client cache-correctness test (a ping triggers exactly one batched refetch per mapped key, unmapped keys stay stable). The **browser-level Playwright two-browser journey** (§05 main-blocking smoke) is **P3-WP09**.

WP08 adds **no** REST endpoint, **no** OpenAPI/contract/migration change, **no** data payload on the socket, **no** new report/feature UI, and **no** Postgres NOTIFY/LISTEN (rejected by N5).

## Scope

### WP08a — ping infrastructure + server coalescing (LL Backend Dev)

1. **Hub — `SpaceInvalidationHub`** (Ledger Infrastructure, mapped from Host `Program.cs` via `app.MapHub<SpaceInvalidationHub>("/hubs/space")`): `[Authorize]` (the existing JwtBearer scheme); `OnConnectedAsync` reads the space id from the query string, **resolves the caller's internal `user_id` via `IIdentityResolver` and validates membership via `ISpaceMembershipQuery`** before `Groups.AddToGroupAsync(connectionId, "space:{spaceId}")` — a non-member connection is **rejected** (aborted), never added (D-P3-SIGNALR-AUTHZ). The client never dictates group membership from an unvalidated id.
2. **Coalescing queue — `SpaceInvalidationQueue` (`ISpaceInvalidationQueue`)**: per-`(spaceId, topic)` dedupe + a channel signal, structurally mirroring `ReportRefreshQueue` (dictionary gate + `Channel<T>` + `ReadBatchAsync(debounceWindow)`); `TryEnqueue(spaceId, topics)` collapses duplicates within the window.
3. **Broadcast service — `InvalidationBroadcastService : BackgroundService`**: reads a coalesced batch after the ~200 ms debounce (`DefaultDebounceWindow = 200 ms`, matching WP12), and for each pending `(spaceId, topic)` sends `spaceInvalidated` to the `space:{spaceId}` group via `IHubContext<SpaceInvalidationHub>`. Registered as an `IHostedService` in `LedgerModule` alongside the existing refresh coalescer.
4. **Enqueue seam**: in `JournalPostingService.ExecuteAsync`, immediately after `_refreshQueue.TryEnqueue(spaceId)` (post-`CommitAsync`), add `_invalidationQueue.TryEnqueue(spaceId, [Topics.TrialBalance, Topics.JournalEntries])`. Fires only on a committed post/reversal; not on rollback, not on an idempotent-replay short-circuit, not on a collision.
5. **Metric — `SpaceInvalidationMetrics`**: a `leafledger.realtime.invalidation.pings` counter (+ enqueued/coalesced counters) via `IMeterFactory`, mirroring `ReportingRefreshMetrics`, so the N5 "≤ ceil(window-count) pings, not N" property is observable.
6. **Azure SignalR config seam (Default mode)**: `AddSignalR()` in Host; when `ConnectionStrings:AzureSignalR` (or `Azure:SignalR:ConnectionString`) is present, chain `.AddAzureSignalR(...)` so production runs through Azure SignalR Service with **no code change** (the app still hosts the `SpaceInvalidationHub`; Azure SignalR proxies). Absent the connection string (local dev / CI / Testcontainers), the self-hosted hub serves directly — no Azure dependency for feature work.
7. **Tests (WP08a)**: hub connect authorized/rejected by membership; cross-space isolation (space-B client never receives a space-A ping); coalescing (N rapid enqueues in one space/topic → ≤ ceil(window) pings); enqueue-only-after-commit (a rolled-back/failed post produces no ping); metric increment.

### WP08b — client invalidation map + batching + exit proof (LL Frontend Dev)

8. **Invalidation map — `app/src/application/query/invalidationMap.ts`**: an explicit `Record<Topic, (spaceId: string) => QueryKey[]>` mapping `'reports.trialBalance' → [qk.reports.trialBalance(spaceId)]`, `'journalEntries.list' → [qk.journalEntries.list(spaceId)]` (the N6 deliverable — the same keys WP06 hard-codes, now centralized and reviewable). Topics are a closed union type; an unknown topic is ignored (logged), never invalidates broadly.
9. **Client hub hook — `app/src/application/query/useSpaceInvalidation.ts`**: owns one `HubConnection` (`@microsoft/signalr`) to `/hubs/space?spaceId=…` with `accessTokenFactory` returning the MSAL bearer (reuse the WP02 token acquisition); on `spaceInvalidated({ spaceId, topic })`, look the topic up in the map and **batch** the invalidations (debounce per key so a burst of pings triggers **one** refetch cycle per key, N5 client half); starts/stops the connection with the component lifecycle; reconnects with backoff. Single connection per shell (not per page).
10. **Shell mount — `app/src/app/AppLayout.tsx` (or a small `LiveInvalidationBoundary`)**: mount `useSpaceInvalidation(VITE_DEMO_SPACE_ID)` once so all pages under the shell live-update; no per-page socket code.
11. **N6 map doc — `app/src/application/query/README.md`** (extend): a "Realtime invalidation map" section — module → topic → query keys, the single documented map N6 requires, covering every mutation in Phase-3 scope (the WP06 post/reverse).
12. **Tests (WP08b)**: the map returns the exact `qk` keys for each topic; the hook invalidates **only** mapped keys on a topic ping and leaves unmapped keys stable (cache-correctness — N6); a burst of pings triggers one batched refetch cycle per key (N5 client half); the hook wires `accessTokenFactory` to the bearer and connects to `/hubs/space` with the space id; unknown topic is a no-op. `@microsoft/signalr` is mocked in unit tests (no live socket).
13. **New dependency**: `@microsoft/signalr` (frontend) + `Microsoft.Azure.SignalR` (backend) — recorded in Dependencies below (plan-file entry per repo rule).

- **No Postgres NOTIFY/LISTEN bridge.** Explicitly rejected by N5 as premature — do not implement.
- **No browser-level Playwright two-browser E2E in this WP.** Playwright infra does not yet exist in this repo (`app/package.json` has no Playwright). - **No real space picker / `GET /spaces`.** Reuse `VITE_DEMO_SPACE_ID` (consistent with WP05/WP06/WP07). The multi-space connection-management design lands with the real picker.
- **No presence / typing indicators / user-list / any hub feature beyond invalidation pings.** The hub does one thing: authorized per-space invalidation broadcast.
- **No new REST endpoint, OpenAPI/TS, or migration change.** The hub is a side channel; `app/src/api/**` and `backend/openapi/**` are unchanged (a regen must produce no diff).
- **No client optimistic-update / socket-driven cache patching.** The socket only invalidates; the server remains the single source of truth (refetch, not push-into-cache).
- **No production Azure SignalR provisioning / IaC.** WP08a adds the *config seam* (`.AddAzureSignalR()` when the connection string is present); provisioning the Azure SignalR resource + wiring the connection string as a deploy secret is an ops/infra task (recorded as a carry-forward), not application code here.

## Decisions (front-loaded, non-accounting — all APPROVED by the user 2026-07-15 on their recommended routes)

Seven decisions, none accounting (transport/cache mechanism; the server owns every value and no value crosses the socket). **All seven were approved on their recommended routes with no overrides**; each is now an implementation constraint for LL Backend Dev (WP08a) / LL Frontend Dev (WP08b).

- **D-P3-SIGNALR-TRANSPORT — the SignalR transport. RECOMMEND: host the `SpaceInvalidationHub` in the app and run Azure SignalR Service in *Default mode* via a `.AddAzureSignalR()` config seam** activated only when a connection string is configured. This satisfies the spec's *"Replace → Azure SignalR Service (scale-out safe)"* mandate for production while keeping local dev / CI / Testcontainers on the self-hosted hub with **no Azure dependency** — the same Hub class serves both; the transport is a config switch, not a code fork.
  - *Alternative A:* Azure SignalR *Serverless* mode (no app-hosted hub; REST/`negotiate` upstream) → the app cannot host `OnConnectedAsync` membership authorization the same way and local dev cannot run the hub without Azure. Rejected (breaks the authorized-group model and local dev).
  - *Alternative B:* self-hosted SignalR only, no Azure seam → repeats the OLD scale-out failure (Part 2 §8). Rejected as the explicit anti-pattern the spec replaces.
- **D-P3-SIGNALR-AUTHZ — group-join authorization (security-critical). RECOMMEND: the hub is `[Authorize]` (existing JwtBearer scheme) and joins a connection to `space:{spaceId}` only after resolving the caller's internal `user_id` (`IIdentityResolver`) and confirming membership (`ISpaceMembershipQuery`) — the identical gate the REST endpoints use (P2-WP06/WP13); a non-member connection is rejected.** Prevents a client from subscribing to another tenant's pings (info-leak: even a data-free ping reveals "space X had activity"). Never trust a client-supplied space id for group membership.
  - *Alternative:* trust the query-string space id and join without a membership check → cross-tenant ping leakage. Rejected (OWASP A01 broken access control).
- **D-P3-SIGNALR-PAYLOAD — ping content. RECOMMEND: the ping carries only `{ spaceId, topic }`** where `topic` is a closed union of the N6 map keys (`reports.trialBalance`, `journalEntries.list`). No entity data, no amounts, no ids beyond the space id. This is the Part 3 §2 mandate (*"no data over the socket"*), pinned as the exact vocabulary.
  - *Alternative:* push changed rows/entry ids to skip the refetch → data on the socket, cache-coherence + authorization complexity, contradicts the spec. Rejected.
- **D-P3-SIGNALR-COALESCE — server coalescing (N5). RECOMMEND: reuse the WP12 coalescing shape verbatim** — a per-`(spaceId, topic)`-deduping `SpaceInvalidationQueue` (`Channel<T>` + gate) drained by an `InvalidationBroadcastService : BackgroundService` on a ~200 ms debounce, enqueued at the existing post-commit seam next to `_refreshQueue.TryEnqueue`. At most one ping per space/topic per window; a metric makes it observable.
  - *Alternative:* broadcast inline per post (no coalescing) → thundering-herd refetch on shared spaces (the exact N5 hazard). Rejected.
- **D-P3-SIGNALR-CLIENT — client structure (N5 client half + N6 map). RECOMMEND: one `useSpaceInvalidation(spaceId)` hook mounted once in the shell, owning a single `HubConnection`, driving an explicit `invalidationMap` (topic → `qk` keys) with batched `invalidateQueries` (one refetch cycle per key per burst).** A single application-layer utility, not per-page socket code (N5's *"single utility, not per-page ad hoc"*).
  - *Alternative:* subscribe per page/component → many connections, duplicate refetches, layering leak into features. Rejected.
- **D-P3-SIGNALR-EXIT — how the two-browser exit journey is verified. RECOMMEND: prove it deterministically inside WP08** — a backend two-`HubConnection` integration test (both clients in one space receive the coalesced ping after a real committed post; a third client in another space does not) + a client cache-correctness test (a ping triggers exactly one batched refetch per mapped key, unmapped keys stable). **Carve the browser-level Playwright two-browser smoke + the Playwright harness into `P3-WP09` (proposed)**, since Playwright is greenfield here and standing it up is its own ≤2-day unit. The Phase-3 exit is **functionally met and proven** by WP08; **browser-certified** (the §05 main-blocking smoke) by WP09.
  - *Alternative:* include Playwright harness + the two-browser journey in WP08b → very likely > 2 days (new E2E toolchain, compose orchestration, hub-over-real-browser flakiness). Rejected for the ≤2-day rule; carved to WP09 instead.
- **D-P3-SIGNALR-DEPS — new dependencies. RECOMMEND: add `@microsoft/signalr` (frontend, the standard SignalR JS client) and `Microsoft.Azure.SignalR` (backend, for the Default-mode seam).** Both are recorded here per the repo's "no new dependency without a plan-file entry" rule. The base ASP.NET Core SignalR hub itself needs no package (shared framework); only `.AddAzureSignalR()` requires the backend package.
  - *Alternative:* a hand-rolled WebSocket protocol → reimplements reconnection/negotiation/backpressure SignalR already provides and diverges from the spec's named technology. Rejected.

## Accounting decisions

**None required — no LL Accounting Expert consult.** WP08 introduces **no** accounting behavior and moves **no** financial value: it is cache-invalidation transport. The ping carries no amounts and no ledger data; the server (SQL views + RLS + the domain) remains the sole authority, and clients merely refetch the already-authoritative REST reports. Two standing boundaries are respected, not decided: money never touches the socket (there is no money on the socket at all), and the report is read-only (the ping triggers a GET refetch, never a write). If, during implementation, anyone proposes putting computed balances or entry data on the socket (a data-over-socket change), **stop** — that contradicts the merged Part 3 §2 verdict and is out of scope.

## Acceptance criteria (concrete, testable)

### WP08a

1. **Authorized hub connect.** A connection to `/hubs/space?spaceId={S}` with a valid bearer whose resolved user is a member of `S` is accepted and added to the `space:{S}` group; an integration test asserts the member connects and is grouped.
2. **Rejected non-member / unauthenticated.** A connection without a valid bearer, or by a non-member of `S`, is rejected (aborted) and never added to the group; a test asserts both rejection paths (401-equivalent + membership denial).
3. **Cross-space isolation (security).** A client in space `A` never receives a ping emitted for space `B`; an integration test with two spaces asserts the space-B ping reaches only space-B connections.
4. **Two-client fan-out.** Two authorized `HubConnection`s in one space both receive `spaceInvalidated` after a single committed post; a test asserts both clients observe the ping (the backend half of the two-browser exit journey).
5. **Ping only after commit.** A rolled-back / validation-failed / idempotent-replay-collision post produces **no** ping; a successful committed post produces one (per topic, subject to coalescing); tests cover a 422 post (no ping) and a 201 post (ping).
6. **Server coalescing (N5).** N rapid committed posts in one space within one window produce **≤ ceil(window-count)** pings per topic, not N; a test drives a burst and asserts the ping/metric count is coalesced (mirrors the WP12 refresh-coalescing test).
7. **No data on the socket.** The `spaceInvalidated` payload contains only `{ spaceId, topic }` (topic from the closed vocabulary) and no ledger/entity data; a test asserts the payload shape.
8. **Azure SignalR seam is config-only.** With no `AzureSignalR` connection string the self-hosted hub serves (CI/Testcontainers pass); the `.AddAzureSignalR()` branch is exercised only when the string is present — a test/inspection confirms the seam is inert by default and adds no Azure dependency to local runs.
9. **Boundary + build.** EF/DB usage stays in Infrastructure; the Host takes no direct EF dependency; NetArchTest boundary tests green; Release build 0/0; the full backend suite (unit + architecture + Testcontainers integration) green.
9b. **Consistent consumer identity across all subject-selection sites (identity remediation).** `SpaceInvalidationHub.TryGetSubjectId` selects the caller subject as `oid ?? sub` for **every** tenant — identical to `HttpContextCurrentUser.SubjectId` — with **no** `ConsumersTenantId` special-case that prefers `sub`; this supersedes D-WP13-SUBJECT (see the P2-WP13 2026-07-15 reversal). A hub integration/unit test asserts that a consumers-tenant connection whose token has a GUID `oid` and a **non-GUID `sub`** resolves to the `oid` (and therefore joins its space group), mirroring `AuthorizationModelTests.Uses_oid_for_the_consumers_tenant_because_the_personal_account_sub_is_not_a_guid`. Any hub test that previously expected `sub` for consumers is updated to expect `oid`, not the reverse.

### WP08b

10. **Invalidation map (N6).** `invalidationMap['reports.trialBalance'](spaceId)` returns `[qk.reports.trialBalance(spaceId)]` and `invalidationMap['journalEntries.list'](spaceId)` returns `[qk.journalEntries.list(spaceId)]`; an unknown topic returns `[]`; a test asserts the exact keys.
11. **Cache-correctness (N6).** On a `spaceInvalidated` ping the hook invalidates **only** the mapped keys for that topic and leaves unmapped keys (e.g. `qk.accounts.list`, `qk.meta`) untouched; a test asserts mapped-key invalidation + unmapped-key stability.
12. **Client batching (N5).** A burst of pings (same topic, within the debounce window) triggers **one** refetch cycle per mapped key, not one per ping; a test asserts a single `invalidateQueries` cycle per key for a burst.
13. **Authenticated single connection.** The hook creates **one** `HubConnection` to `/hubs/space` carrying the space id, with `accessTokenFactory` returning the MSAL bearer (reuse WP02); a test asserts the URL, the space id, and that the token factory is wired; the connection starts/stops with the component and is not duplicated per page.
14. **Shell-mounted, layering-clean.** `useSpaceInvalidation` is mounted once under `AppLayout`; feature pages contain no socket code and no `@microsoft/signalr` import; the hook lives in `application`; ESLint layering/leaf gates green.
15. **Exit journey proven (D-P3-SIGNALR-EXIT).** Together, AC4 (backend two-client fan-out after a real committed post) + AC11/AC12 (client refetches exactly the mapped keys, batched) deterministically prove "two browsers in one space live-update"; the plan records that the browser-level Playwright certification is P3-WP09.
16. **No out-of-scope surface / no contract drift.** No REST endpoint, OpenAPI/TS, or migration change (`app/src/api/**` + `backend/openapi/**` byte-unchanged; a regen produces no diff); no data on the socket; no NOTIFY/LISTEN; no new report/feature UI; a scope scan confirms the diff is limited to the file list.
17. **All gates green.** Frontend: `npm run lint`, `npm run typecheck`, `npm test`, `npm run check:page-budget`, `npm run build`, duplicate-key check, and `npm audit --omit=dev` (no new vulnerabilities from `@microsoft/signalr`) green. Backend: Release build + full suite green.

## Boundary note

WP08 respects **features → application → api** on the client (the `HubConnection` + invalidation map live in `application/query`; features never import `@microsoft/signalr` or touch the socket) and the module boundary on the server (the hub, queue, broadcast service, and metrics live in Ledger Infrastructure; the Host only maps the hub and configures the transport; no Host EF dependency). No money crosses the socket (no money on the socket at all); the ping is data-free and triggers a REST refetch, so the server (domain + SQL views + RLS) stays the single source of truth. Group-join is server-authorized by the same membership gate as the REST endpoints (no cross-tenant leakage).
