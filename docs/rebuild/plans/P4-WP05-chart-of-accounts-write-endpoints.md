## Backend design (WP05a — the CoA write endpoints)

**No migration.** Every schema object the writes need already exists (P2-WP02). WP05a adds a write service, endpoints, one permission, and the OpenAPI/TS regen.

- **Write service (Ledger Infrastructure) — `IAccountManagementService` + `AccountManagementService`:** mirrors the `JournalPostingService.ExecuteAsync` shape — open connection, begin transaction, `BindTransactionAsync(connection, txn, spaceId, actorId)` (principal-bound RLS + `app.current_actor` for the audit triggers), enforce/replay/store idempotency inside the transaction (advisory lock + `idempotency_keys`), execute the command, `SaveChanges`/SQL, commit, enqueue space invalidation. Each command **validates through the ChartOfAccounts domain first** (`AccountCodeRange.Create`, `Account.Create`, `CurrencyPolicyEvaluator`, `AccountGroup`), returning structured 422 on invariant failure, then persists. Methods (M1):
  - `CreateAccountAsync(spaceId, actor, idemKey, CreateAccountCommand)` — validate name/currency/kind/validity + **code within the target group's `code_range`** + currency-policy-for-kind; insert; `UNIQUE(space_id, code)` violation → `422 account.code_taken`; group-range violation → `422 account.code_out_of_group_range`.
  - `UpdateAccountAsync(…, accountId, UpdateAccountCommand)` — edit only the fields allowed by **C-P4-COA-EDIT**; after the first posted line, `code`, `currency`, `kind`, `valid_from`, `valid_to`, `fx_policy`, and `group_id` are immutable, while `name` remains editable with audit history. Editing an immutable field on a posted account → `422 account.field_immutable_after_posting`.
  - `SetAccountActiveAsync(…, accountId, bool active)` — activate/deactivate (`is_active`); deactivation does not touch history and is always reversible (re-activate). (Posting-validity already rejects inactive accounts per P2-WP04, so deactivation is the M1 substitute for delete on posted accounts.)
  - `CreateAccountGroupAsync(…, CreateGroupCommand)` — validate name + `AccountCodeRange`; insert; the `account_groups_no_overlap` GiST exclusion violation → `422 group.code_range_overlap`.
  - `UpdateAccountGroupAsync(…, groupId, UpdateGroupCommand)` — edit name / code range (a range edit that would exclude an existing member account's code → `422 group.range_excludes_member`; a range edit overlapping a sibling → `422 group.code_range_overlap` from the GiST wall). Group name remains mutable; range edits must retain every existing member account code.
- **Idempotency (D-P4-COA-IDEM):** all five writes require an `Idempotency-Key` (ULID) header, enforced with the **same** `TryGetIdempotencyKey` validation (`400 idempotency.key_required`/`key_invalid`), request-hash replay, and advisory lock as posting — reusing `IdempotencyStore` + the `ExecuteAsync` envelope (generalized so `target` distinguishes `account.create`/`account.update`/… from `post`). Retried same key+payload → the original 2xx replayed with `Idempotent-Replayed: true`; same key+different payload → `409 idempotency.key_reused`.
- **Contracts (`Application/Accounts/AccountManagementContracts.cs`, `AccountCatalogContracts` style):** `CreateAccountCommand`, `UpdateAccountCommand`, `SetAccountActiveCommand`, `CreateGroupCommand`, `UpdateGroupCommand`, and thin response records (`AccountView` reused for account responses; a `GroupView(Guid Id, string Name, int RangeStart, int RangeEnd, Guid? ParentId, string? FxPolicy)` added for groups + a `GET …/groups` read to back the management UI's group picker — see below).
- **Group read (small, additive):** `GET /api/v1/spaces/{spaceId}/groups` → `GroupCatalogReport` (RLS-bound, `ledger.read`) so the account form's group selector and the groups management table have a source. Mirrors the P3-WP05 read shape.
- **Routes (extend `LedgerEndpoints`, existing `configureAuthorization` seam):**
  - `POST   …/accounts` → 201 `AccountView` (`accounts.manage`, Idempotency-Key, 400/409/422/401/403).
  - `PATCH  …/accounts/{accountId:guid}` → 200 `AccountView` (`accounts.manage`, idempotent).
  - `POST   …/accounts/{accountId:guid}/activate` and `…/deactivate` → 200 `AccountView` (`accounts.manage`, idempotent). *(Sub-resource actions rather than a PATCH-`is_active`, for clean audit/idempotency semantics — D-P4-COA-VERBS.)*
  - `GET    …/groups` → 200 `GroupCatalogReport` (`ledger.read`).
  - `POST   …/groups` → 201 `GroupView` (`accounts.manage`, idempotent).
  - `PATCH  …/groups/{groupId:guid}` → 200 `GroupView` (`accounts.manage`, idempotent).
  All 422s use stable ProblemDetails codes + `extensions["errors"]` `{ code, message, field? }`, exactly like posting.
- **Permission (`ModulePermissions`) — new `accounts.manage` (D-P4-COA-PERM):** granted to **Owner + Admin** only (chart maintenance is an administrative, not everyday-posting, capability). Members/Viewers keep read-only access to accounts/groups via `ledger.read`. Extend `Allows` + the WP06 authorization tests.
- **OpenAPI/TS regeneration (REQUIRED):** rebuild `backend/openapi/leafledger-v1.json` + regenerate `app/src/api/schema.d.ts`; the CI `contract` gate must be green. A regen diff here is **expected and required**.
- **Register `IAccountManagementService` (+ the group read service if separate) in `LedgerModule.cs`.**
- **No EF entity/model change → `HasPendingModelChanges()` stays green** (writes are parameterized SQL/existing entities under the bound transaction, same as the read/post services). No schema, no grant, no audit-trigger change (triggers already fire on both tables).

## Goal

An Owner/Admin can **manage their chart of accounts from the app** — create a new account (code, name, currency, kind, group, validity), edit its mutable fields, activate/deactivate it, and create/edit custom groups with code ranges — every write **server-authoritative**, idempotent, RLS-scoped, audited, and validated through the existing ChartOfAccounts domain, with structured 422s rendered inline. Concretely: WP05a exposes the CoA write endpoints (`POST/PATCH …/accounts`, activate/deactivate, `GET/POST/PATCH …/groups`) under a new `accounts.manage` permission with idempotency + OpenAPI/TS regen; WP05b renders the management UI (forms/modals + activate-deactivate + groups table) over TanStack Query mutations with UX-only mirrored validation and live invalidation.

This WP adds **no** account import/export (P4-WP06), **no** account ownership/attribution editing (P4-WP11), **no** normative built-in starter chart, **no** account/group hard-delete, and **no** client-side role/rights table.

## Accounting decisions

**C-P4-COA-EDIT — post-posting mutability & lifecycle — RESOLVED 2026-07-22 by LL Accounting Expert.** This is a target control policy, not an OLD behavior claim. After the first posted journal line, the following account fields are immutable: **`code`, `currency`, `kind`, `valid_from`, `valid_to`, `fx_policy`, and `group_id`**. This prevents historical re-bucketing, currency/evidence inconsistency, eligibility-window changes, FX-policy rewrites, and group reclassification. **`name` remains mutable** with the existing audit trigger preserving the prior value. Before the first posted line, these fields remain editable subject to the domain and database invariants.

**Group policy:** `name` remains mutable with audit history. `code_range` remains editable only when the new range is non-overlapping and still contains every existing member account code; narrowing a range around an existing account is rejected as `422 group.range_excludes_member`. The existing GiST exclusion remains the second wall for sibling-range overlap.

**Lifecycle:** accounts are **deactivate-only** in this WP; no hard delete. Deactivation is reversible and does not alter history. Groups are not deleted in this WP; if group deletion is introduced later, a non-empty group must be rejected with `group.not_empty`. Existing posted rows remain intact and foreign-key history is preserved.

The resulting rules must be pinned by explicit integration tests: each immutable-after-posting account field returns `422 account.field_immutable_after_posting`; name updates remain successful and audited; valid group range edits succeed; member-excluding edits return `422 group.range_excludes_member`; overlapping ranges return `422 group.code_range_overlap`; and no account/group hard-delete route is exposed. **No golden fixtures are required.** Route the policy to LL Docs Editor for a candidate ADR before implementation if the architecture log is to carry this target control explicitly.

**Standing guards (load-bearing):** the **structural** invariants (code unique/in-range, range non-overlap, `valid_from ≤ valid_to`, currency-policy-per-kind) are **not** re-decided here — they are the already-approved P2-WP03 domain + P2-WP01 fixtures and are simply enforced by reusing the domain factories. Money stays integer minor units + ISO currency. The client mirrors server validation for UX only; the server + DB constraints are the wall. If any write rule cannot be traced to the P2-WP03 domain, an existing fixture, or a C-P4-COA-EDIT answer, **stop and route to LL Accounting Expert**.

## Acceptance criteria (concrete, testable)

1. **Create account.** `POST …/accounts` with a valid command persists the account (RLS-scoped, audited) and returns 201 `AccountView`; an integration test asserts the round-trip and that the audit trigger wrote a row.
2. **Structural validation (reused domain).** Duplicate code → `422 account.code_taken`; code outside the target group's `code_range` → `422 account.code_out_of_group_range`; unsupported currency → `422 currency.unsupported` through `Account.Create`. Currency-policy matching remains a posting-time transaction-line check in the posting service and is covered by the existing P2-WP01 fixtures; account creation does not invent a currency restriction by kind.
3. **Edit mutability (C-P4-COA-EDIT).** After an account has a posted line, editing each of `code`, `currency`, `kind`, `valid_from`, `valid_to`, `fx_policy`, and `group_id` via `PATCH …/accounts/{id}` returns `422 account.field_immutable_after_posting`; editing `name` succeeds and writes an audit row. Before posting, valid changes remain possible subject to the domain and database invariants.
4. **Activate/deactivate.** `POST …/accounts/{id}/deactivate` sets `is_active=false` (a subsequent post referencing it is rejected by posting-validity); `…/activate` restores it and a post then succeeds; idempotent (re-deactivating is a no-op success).
5. **Groups.** `POST …/groups` creates a custom group with a code range; an overlapping range → `422 group.code_range_overlap` (GiST exclusion surfaced as a structured 422, never a raw 500); `PATCH …/groups/{id}` editing a range that excludes an existing member's code → `422 group.range_excludes_member`; `GET …/groups` returns RLS-scoped groups.
6. **Authorization (`accounts.manage`).** Every write is `accounts.manage`-gated: anonymous → 401, a Member without the permission → 403, Owner/Admin → 2xx; reads (`…/groups`, `…/accounts`) stay `ledger.read` (Viewer → 200). Cross-space writes are blocked by the RLS second wall (a second space's data is untouched).
7. **Idempotency (non-negotiable #4).** All writes require a valid ULID `Idempotency-Key` (`400 idempotency.key_required`/`key_invalid` otherwise); a retried create with the same key+payload replays exactly one row with `Idempotent-Replayed: true`; same key + different payload → `409 idempotency.key_reused`.
8. **No schema/model drift.** No EF entity/model change, no new migration/grant; `HasPendingModelChanges()` no-drift test stays green.
9. **Contract regenerated and gate-green.** `backend/openapi/leafledger-v1.json` + `app/src/api/schema.d.ts` regenerated and committed; CI `contract` gate green; the seven operations + new schemas present.
10. **Management UI.** An Owner/Admin can create/edit an account and create/edit a group through modal forms rendering shared primitives, activate/deactivate a row, and see inline server 422 messages; loading/submitting states present; each page/modal ≤ 450 lines (page budget green) — proven by component tests.
